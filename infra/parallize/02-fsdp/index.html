Module 2 · FSDP / ZeRO

把模型状态也切开：ZeRO 与 FSDP

DP 让每张卡冗余存了 N 份模型状态。ZeRO 的洞察是：这些状态在每张卡上各存一份纯属浪费——把优化器状态、梯度、参数依次切到各卡，用时再临时聚合。ZeRO-3 就是 FSDP。

📦 ZeRO 1/2/3 三阶段 🔄 all-gather 参数 + reduce-scatter 梯度 📉 显存降到 1/N 🟠 torchtitan FSDP2 🟢 Megatron DistributedOptimizer

01动机：DP 的冗余

上一章的结论：纯 DDP 下每张卡都存完整的参数（$2P$）、梯度（$2P$）、优化器状态（$12P$），共 $16P$ 字节，完全不随卡数下降。但仔细想——这 N 份副本内容完全相同，为什么每张卡都要存一整份？

ZeRO（Zero Redundancy Optimizer）的答案：不存了，切开。每张卡只持有 $1/N$ 的模型状态，需要完整参数时临时用 all-gather 拼起来，用完即丢。这样显存随卡数线性下降，代价只是多一点通信。

02ZeRO 三阶段：一刀一刀往下切

按「切得越来越多」排序，先切最大的那块（优化器状态占了 75%）：

阶段切什么每卡模型状态显存对应实现
baseline DDP啥都不切$2P + 2P + 12P = 16P$普通 DP
ZeRO-1切优化器状态$2P + 2P + 12P/N$Megatron DistributedOptimizer
ZeRO-2再切梯度$2P + (2P + 12P)/N$—
ZeRO-3再切参数$(2P+2P+12P)/N = 16P/N$FSDP（torchtitan FSDP2）
ZeRO 一刀一刀往下切，N 越大单卡省得越多。Megatron 的分布式优化器停在 ZeRO-1，FSDP 直接做到 ZeRO-3。

03FSDP（=ZeRO-3）的前向/反向通信流

FSDP 的核心节奏：参数平时是切碎的，用到哪层才把那层 all-gather 成完整，算完立刻丢回切片态。

FSDP 一个 FSDP unit（通常一层）的通信序列：前向 all-gather→算→reshard，反向 all-gather→算→reduce-scatter。

04torchtitan：FSDP2 = per-parameter DTensor 分片

torchtitan 用 PyTorch 原生的 fully_shard（FSDP2）。它逐个 TransformerBlock 调用 fully_shard，每个 block 成为一个独立的通信单元（all-gather/reduce-scatter 的粒度），最后对整个 model 再包一层：

torchtitan · models/llama3/parallelize.py:163（apply_fsdp 核心）
mp_policy = MixedPrecisionPolicy(
    param_dtype=param_dtype,       # 计算用 bf16
    reduce_dtype=reduce_dtype,     # 梯度归约用 fp32
    cast_forward_inputs=False,
)
fsdp_config = {"mesh": dp_mesh, "mp_policy": mp_policy}
reshard_after_forward = get_fsdp_reshard_after_forward_policy(reshard_after_forward_policy, pp_enabled)

for layer_id, transformer_block in model.layers.items():
    fully_shard(transformer_block, **fsdp_config, reshard_after_forward=reshard_after_forward)
fully_shard(model, **fsdp_config)

reshard_after_forward 控制「前向后是否立刻丢弃完整参数」——用通信换显存。PP 场景下默认不 reshard，避免每个 micro-batch 重复 all-gather：

torchtitan · distributed/fsdp.py:28
match reshard_after_forward_policy:
    case "always": return True
    case "never":  return False
    case "default":
        # PP 下不 reshard，避免每 micro-batch 重复且无法重叠的 all-gather
        return not pp_enabled

💡 FSDP2 vs FSDP1 FSDP1 把一整个单元的参数压成一个大 flat buffer 再切；FSDP2 是 per-parameter 分片——每个参数独立成为沿 dim-0 切分的 DTensor（distribute_tensor(param, mesh, [Shard(0)])）。这让混合精度、reshard_after_forward、与 TP/PP/EP 的组合都更干净，也是 torchtitan 选 FSDP2 的原因。

05Megatron：DistributedOptimizer 停在 ZeRO-1

Megatron 的路线不同：它不切常驻参数，只把优化器状态按 DP rank 切分（ZeRO-1），梯度走 reduce-scatter。先看它如何把梯度 buffer 平均切给每个 rank：

Megatron-LM · optimizer/distrib_optimizer.py:188（_build_model_gbuf_range）
gbuf_size = bucket.grad_data.numel()
max_gbuf_range_size = gbuf_size // data_parallel_world_size   # 平均切成 N 段
# 第 r 段归 rank r “拥有”：它只 reduce 这段梯度、只为这段建 fp32 master + Adam 状态
gbuf_world_range = gbuf_world_all_ranges[data_parallel_rank]

每个 rank 只为自己那段建 fp32 master 参数和优化器状态——这就是优化器状态降到 $1/N$ 的地方：

Megatron-LM · optimizer/distrib_optimizer.py:388（只为本 rank 的 shard 建 master）
shard_model_param = model_param.detach().view(-1)[param_range.start : param_range.end]
shard_main_param  = shard_model_param.clone().float()   # 只覆盖本 rank 的 shard → 状态 1/N

梯度归约用 reduce-scatter（每 rank 只收到自己 shard 的梯度），更新后再 all-gather 把完整参数拼回供下次前向：

Megatron-LM · distributed/param_and_grad_buffer.py:617 / :420
# 反向：开启 distributed optimizer → reduce-scatter（否则退化成 DDP 的 all-reduce）
grad_reduce_handle = dist_reduce_scatter_func(local_data_view, bucket.grad_data, ...)
# step 后：每 rank 只更新了自己 shard 的参数，再 all-gather 拼回完整 param_data
dist_all_gather_func(bucket.param_data, local_data_view, group=..., async_op=async_op)

⚠️ 关键区别 Megatron DistributedOptimizer 保留完整参数副本（只切优化器状态），所以它是 ZeRO-1，不等价于 FSDP/ZeRO-3。它通信更省，但显存降幅小于 FSDP（少降了参数和梯度那两块 $2P$）。

06通信量：为什么 FSDP ≈ DP 的 1.5×

回忆 M0 的恒等式 all-reduce = reduce-scatter + all-gather，每个的通信量都是 $P$ 量级：

  • DDP：每 step 一次梯度 all-reduce ≈ $2P$（reduce-scatter $P$ + all-gather $P$）。
  • FSDP / ZeRO-3：前向 all-gather 参数（$P$）+ 反向 all-gather 参数（$P$）+ 反向 reduce-scatter 梯度（$P$）= $3P$。

$3P / 2P = 1.5$。多出的那 0.5× 是前向那次额外的参数 all-gather，换来的是参数、梯度、优化器状态全部降到 $1/N$。在显存吃紧时这笔交易非常划算——而且这些 all-gather 可以靠预取（prefetch）与计算重叠，实际墙钟开销远小于 1.5×。

07FSDP ≡ ZeRO-3

两者算法完全等价：参数、梯度、优化器状态全部沿 DP 维切成 $1/N$；用到完整参数时临时 all-gather、用完丢弃；梯度 reduce-scatter 后各管各的 shard。差异只在实现：

分片粒度参数 flat partition + hookper-parameter DTensor（Shard(0)）
通信单元参数分组fully_shard(module) 决定
混合精度ZeRO configMixedPrecisionPolicy

08对比与启用

torchtitan FSDP2（ZeRO-3）

  • 切参数 + 梯度 + 优化器状态 → $16P/N$
  • per-parameter DTensor 分片
  • --parallelism.data_parallel_shard_degree N
  • 配 replicate_degree>1 = HSDP

Megatron DistributedOptimizer（ZeRO-1）

  • 只切优化器状态 → $4P + 12P/N$
  • 梯度 reduce-scatter，参数仍完整
  • --use-distributed-optimizer
  • --overlap-grad-reduce / --overlap-param-gather

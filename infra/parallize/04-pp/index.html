Module 4 · 流水线并行

流水线并行 PP：把不同层放到不同卡

模型层数太多放不下，就把它沿深度切成几段（stage），每张卡负责几层，像工厂流水线一样接力。难点只有一个词：气泡——以及一连串为消灭气泡而生的调度算法。

🫧 气泡 (P-1)/(P-1+M) 📦 GPipe → 1F1B → 交错 → Zero-Bubble ↔️ p2p send/recv 🔥 warmup/steady/cooldown

01核心思想：沿深度切

TP 是把一层横着切，PP（Pipeline Parallel）是把层与层之间竖着切：模型的前几层放 stage 0，中间几层放 stage 1……最后几层放 stage P-1。数据像流水线一样，前向时激活从 stage 0 一路传到 stage P-1，反向时梯度逆序传回。

torchtitan 里能清楚看到这个切分：embedding 放首 stage，transformer 层平摊，norm+lm_head 放尾 stage：

torchtitan · distributed/pipeline_parallel.py:346（_generate_llm_fqn_per_model_part）
num_effective_layers = num_layers + input_weight + output_weight
layers_per_stage = num_effective_layers // num_stages
# stage 0 放 tok_embeddings + 前若干层；最后 stage 放尾部层 + norm + lm_head
if stage_idx == 0:
    stage_modules.append("tok_embeddings")
elif stage_idx == num_stages - 1:
    stage_modules.extend(["norm", "lm_head"])

跨 stage 只在边界传一次激活/梯度（点对点通信），通信量极小——这正是 PP 能跨机的原因（对比 TP 每层都通信）。

02气泡：PP 的原罪

流水线是链式依赖：stage 1 必须等 stage 0 把激活传过来才能开算。如果一个 batch 整块往里灌，启动阶段后面的 stage 都在干等，排空阶段前面的 stage 又在干等。这些空闲就是气泡（bubble）。

解药：把一个 batch 拆成 $M$ 个 micro-batch 接连灌入，让各 stage 尽快都忙起来。理想气泡占比约为：

$$ \text{bubble ratio} \approx \frac{P-1}{P-1+M}, \qquad P=\text{stage 数},\ M=\text{micro-batch 数} $$

所以 $M$ 越大于 $P$，气泡越小。torchtitan 在 micro-batch 数不足时会直接告警：

torchtitan · distributed/pipeline_parallel.py:268
if n_microbatches < num_total_stages:
    logger.warning("... may result in a bubble in the pipeline.")
GPipe vs 1F1B（4 stage 示意）：两者气泡总量相同，但 1F1B 把激活显存从 ∝M 降到 ∝P。

03调度演进：一条消灭气泡的主线

调度关键改进代价
GPipe拆 micro-batch 填满流水线，气泡从 $(P-1)/P$ 降到 $(P-1)/(P-1+M)$同时驻留 $M$ 份激活，显存 ∝ M
1F1B稳态「做 1 个前向就立刻做 1 个反向」，在飞激活上限 ≈ P，显存与 M 解耦气泡总量同 GPipe
交错 1F1B每 rank 持 num_model_chunks 个不连续层块（虚拟 stage），气泡 ∝ 1/chunksp2p 通信次数翻倍
Zero-Bubble把反向拆成「激活梯度 B」+「权重梯度 W」两步，用 V 形布局填补气泡，逼近 0调度复杂、实现精细

Megatron 用一个分发函数按 PP 度数和虚拟 stage 自动选调度：

Megatron-LM · pipeline_parallel/schedules.py:142（get_forward_backward_func）
if pp_size > 1:
    if vp_size is not None:
        forward_backward_func = forward_backward_pipelining_with_interleaving       # 交错 1F1B
    else:
        forward_backward_func = forward_backward_pipelining_without_interleaving    # 1F1B
else:
    forward_backward_func = forward_backward_no_pipelining

041F1B 的三段：warmup / steady / cooldown

1F1B 一个 step 分三段。warmup：只做前向，把流水线填满；steady：一前一后交替，是稳态吞吐区；cooldown：只做剩余反向，排空。warmup 的 micro-batch 数取决于本 stage 离末端多远：

Megatron-LM · pipeline_parallel/schedules.py:2166
# 越靠前的 stage 预热越多；最后一个 stage 预热 0
num_warmup_microbatches = total_stages - current_stage - 1
num_warmup_microbatches = min(num_warmup_microbatches, num_microbatches)
num_microbatches_remaining = num_microbatches - num_warmup_microbatches

稳态循环是 1F1B 的灵魂——一次融合通信「发前向激活、同时收回反向梯度」，紧接着立刻做一个反向：

Megatron-LM · pipeline_parallel/schedules.py:2269（steady 阶段）
for i in range(num_microbatches_remaining):
    output_tensor, _ = forward_step(...)                                   # 1 个 F
    output_tensor_grad = p2p_communicator.send_forward_recv_backward(...)  # 发激活 + 收梯度
    input_tensor  = input_tensors.pop(0)                                   # 取最早那对
    output_tensor = output_tensors.pop(0)
    input_tensor_grad = backward_func(input_tensor, output_tensor, output_tensor_grad, config)  # 1 个 B
    input_tensor = p2p_communicator.send_backward_recv_forward(...)        # 发梯度 + 收下一激活

05点对点通信：stage 之间怎么传

PP 的通信不是集合通信，而是点对点 send/recv。Megatron 的 P2PCommunicator 封装四个方向 + 两个稳态融合算子，底层用 torch.distributed.P2POp + batch_isend_irecv：

Megatron-LM · pipeline_parallel/p2p_communication.py:424 / :486
def recv_forward(...):   # 从上一 stage 收激活；首 stage 收到 None
    if is_first_stage: input_tensor = None
    else: input_tensor, _, _ = self._communicate(recv_prev=True, recv_next=False, ...)

def send_forward(...):   # 把激活发给下一 stage；尾 stage 不发
    if not is_last_stage: self._communicate(tensor_send_next=output_tensor, ...)
# 另有 recv_backward / send_backward，以及稳态用的 send_forward_recv_backward / send_backward_recv_forward

06torchtitan：复用 PyTorch 的 pipelining

torchtitan 不自己写调度，而是直接用 PyTorch 的 torch.distributed.pipelining，把 schedule 名字解析成对应的类（GPipe / 1F1B / Interleaved1F1B / ZBVZeroBubble / DualPipeV）：

torchtitan · distributed/pipeline_parallel.py:253（_build_pipeline_schedule）
schedule_class = get_schedule_class(parallelism.pipeline_parallel_schedule)   # 名字 → 类
looped_schedule = issubclass(schedule_class, PipelineScheduleMulti)
n_microbatches = batch_size // microbatch_size
if looped_schedule:   # 多 stage（交错 / zero-bubble）
    schedule = schedule_class(stages, n_microbatches=n_microbatches, loss_fn=loss_fn, ...)
else:                 # 单 stage（GPipe / 1F1B）
    schedule = schedule_class(stages[0], n_microbatches=n_microbatches, loss_fn=loss_fn, ...)

切模型时它真的把不属于本 stage 的子模块删掉/置 None，得到一个「瘦身版」模型：

torchtitan · distributed/pipeline_parallel.py:444（_split_module）
model = copy.deepcopy(whole_model)
for module_name, module_value in model.named_children():
    if isinstance(module_value, (nn.ModuleDict, nn.ModuleList, ...)):
        new_layers = ModuleList([layer for i, layer in enumerate(module_value) if i in indices_to_keep])
        setattr(model, module_name, new_layers)        # 只留本 stage 的层
    elif module_name not in modules_to_keep:
        setattr(model, module_name, None)              # 不属于本 stage 的整模块置 None

训练循环里只有首 stage 喂输入、尾 stage 收 loss，中间 stage 全由 schedule 的 step() 驱动：

torchtitan · trainer.py:661
if self.pp_has_first_stage:
    self.pp_schedule.step(inputs, target=targets, losses=losses, ...)
else:
    self.pp_schedule.step(target=targets, losses=losses, ...)
if self.pp_has_last_stage:                              # 只有最后 stage 有 loss
    loss = torch.sum(torch.stack(losses)).to(self.device)

07对比与启用

Megatron-LM 手写调度

  • get_forward_backward_func 按 pp/vp 自动选
  • 1F1B / 交错 1F1B 显式 warmup/steady/cooldown
  • --pipeline-model-parallel-size
  • --num-layers-per-virtual-pipeline-stage（交错）

torchtitan 复用 pipelining

  • torch.distributed.pipelining 的 schedule 类
  • _split_module 删层切 stage
  • pipeline_parallel_degree + pipeline_parallel_schedule
  • 支持 GPipe / 1F1B / Interleaved1F1B / ZBVZeroBubble

Module 4 · 流水线并行

流水线并行 PP：把模型像流水线一样接力

模型层数太多、一张卡放不下，就把它沿深度切成几段，每张卡负责几层，像工厂流水线那样一棒接一棒。本章带你从「为什么会有气泡」一步步走到「1F1B 怎么把气泡和显存同时压下来」。

🏭 流水线比喻 🫧 气泡 = GPU 干等 📦 micro-batch 填满流水线 🔁 1F1B 省激活显存

01先建立直觉：一条工厂流水线

想象一条汽车流水线有 4 个工位：装底盘 → 装发动机 → 装车身 → 喷漆。一辆车必须依次经过这 4 个工位。流水线并行（Pipeline Parallel, PP）就是把模型的层当作工位，把数据当作要加工的车。

具体怎么切？假设模型有 32 层、你有 4 张 GPU，那就每张卡放 8 层:

把 32 层切成 4 段（stage），每张卡管 8 层。卡之间只在边界传一个激活/梯度张量。

注意两个要点：

  • 前向时，stage 0 算完它的 8 层，把一个激活张量传给 stage 1；stage 1 接着算，再传给 stage 2……最后 stage 3 算出 loss。反向时梯度沿原路逆向传回。
  • 卡之间只在 stage 边界传一个张量（不是每层都传），通信量极小。这就是 PP 和上一章 TP 的最大区别——TP 每层都要 all-reduce，所以只能在机内；PP 通信稀疏，可以跨机器。

torchtitan 里能看到这种切分：embedding 放首 stage，transformer 层平摊，最后的 norm+lm_head 放尾 stage:

torchtitan · distributed/pipeline_parallel.py:346（自动决定每个 stage 放哪些模块）
layers_per_stage = num_effective_layers // num_stages
if stage_idx == 0:
    stage_modules.append("tok_embeddings")        # 首 stage 放 embedding
elif stage_idx == num_stages - 1:
    stage_modules.extend(["norm", "lm_head"])      # 尾 stage 放输出层

02朴素做法的问题：流水线空着一大半

切好了，怎么跑？最直接的想法：把一整个 batch 灌进 stage 0，算完传给 stage 1……但你马上会发现一个尴尬的事实：同一时刻只有一个工位在干活，其它三个都在干等。

stage 0 在忙的时候，stage 1/2/3 还没收到数据，只能空转；等数据传到 stage 3 时，stage 0/1/2 又没事干了。4 张昂贵的 GPU，利用率只有 $1/4$:

前向 F反向 B气泡（GPU 干等）
横轴是时间，每行一张卡，每格是一个时间片。大片空白 = 气泡。4 张卡，大部分时间在空转。

⚠️ 这就是 PP 的原罪：气泡（bubble） 流水线是链式依赖——后一个工位必须等前一个工位的产出。启动时后面的 stage 在等（叫 warmup 气泡），收尾时前面的 stage 在等（叫 cooldown 气泡）。这些「GPU 干等」的空白就是气泡。接下来三节，都是在想办法把这片空白填满。

03GPipe：把一个 batch 拆成小份，填满流水线

回到工厂的比喻：如果你一次只往流水线里放一辆车，那确实大部分工位在闲着。但如果你一辆接一辆连续地放车呢？很快 4 个工位就都各自忙着不同的车了。

GPipe 就是这个思路：把一个大 batch 切成 $M$ 个 micro-batch（小份），一份接一份连续地喂进流水线。当 micro-batch 1 走到 stage 1 时，micro-batch 2 正好进入 stage 0——流水线被填满了:

前向 Fi反向 Bi气泡
F1…F6 是 6 个 micro-batch 的前向，B6…B1 是它们的反向。中间的空白小多了，但仔细看：还剩两块三角形气泡。

气泡还在，但小多了——能算出来

填满流水线需要「预热」$P-1$ 步（最后一个 stage 要等前面都传过来），排空也要 $P-1$ 步。所以无论多少 micro-batch，头尾各有一段固定的气泡。气泡占比可以估算:

$$ \text{气泡占比} \approx \frac{P-1}{M + P - 1}, \qquad P=\text{stage 数},\ M=\text{micro-batch 数} $$

代入感受一下（$P=4$）:

micro-batch 数 M气泡占比含义
1（朴素）3/4 = 75%几乎全在空转
63/9 ≈ 33%还有三分之一浪费
323/35 ≈ 8.6%基本填满了

结论很直观:micro-batch 数 $M$ 越大于 stage 数 $P$，气泡越小。torchtitan 在 $M$ 不够时会直接告警:

torchtitan · distributed/pipeline_parallel.py:268
if n_microbatches < num_total_stages:
    logger.warning("... may result in a bubble in the pipeline.")

⚠️ GPipe 的新麻烦：显存爆了 想把气泡降到 8%，就得用 $M=32$。但 GPipe 是「先做完所有前向，再做所有反向」——这意味着 32 个 micro-batch 的激活全都要留着，直到反向才用得上。看上图 S0 那一行:它存了 6 份激活（F1–F6）一直挂到反向。激活显存 ∝ M，M 一大就 OOM。我们陷入两难:M 小则气泡大，M 大则显存爆。

041F1B：让反向尽早开始，把显存压回去

1F1B 的洞察非常漂亮:你根本不需要等所有前向都做完。 一旦 micro-batch 1 的前向走到最后一个 stage，它的反向就可以立刻开始往回走——反向一做完，这份激活就能马上释放，不必一直占着显存。

于是调度变成:每个 stage 在稳定期里「做一个前向，紧接着做一个反向」，交替进行（这就是名字 1F1B = One Forward One Backward 的由来）:

前向 Fi反向 Bi气泡
看最后一行 S3：它一拿到 F1 就立刻做 B1，然后 F2、B2…… 一前一后咬合得严丝合缝。反向被尽量提前了。

🔑 1F1B 的精髓:气泡没变，但显存变小了

对比上面两张图你会发现一个反直觉的事实——1F1B 和 GPipe 的总时长（气泡总量）完全一样（都是 18 个时间片）。1F1B 并没有让流水线更快。它真正改变的是显存:因为反向被尽早触发、激活用完即释放，每个 stage 同时「挂着」的激活数量从 $M$ 降到了约 $P$:

方案S0 最多同时保存几份激活激活显存
GPipe（M=6）6 份（= M）∝ M，M 大就 OOM
1F1B（M=6）4 份（≈ stage 数 P）∝ P，与 M 解耦

这就解开了 GPipe 的两难:既然显存不再随 $M$ 增长，你就可以放心用很大的 $M$ 来把气泡压到很小，而不必担心 OOM。这正是 1F1B 成为事实标准的原因。

05看懂 1F1B：warmup / steady / cooldown 三段

把上面 1F1B 图里任意一行拆开看，每个 stage 都经历三个阶段:

  1. warmup（预热，只做前向）:先连做几个前向，把流水线填起来。离输入越近的 stage 预热越多——它得先攒够「在飞」的 micro-batch，下游才有活干。预热数 = P - 1 - stage_id（最后一个 stage 预热 0，拿到第一个前向就能立刻反向）。
  2. steady（稳态，1F1B 交替）:做一个前向、紧接一个反向，循环。这是流水线满载、效率最高的区间（图里中间咬合得最紧的部分）。
  3. cooldown（排空，只做反向）:前向都喂完了，把流水线里剩下的反向逐个做完。

Megatron 把这套逻辑显式写出来。预热数的计算:

Megatron-LM · pipeline_parallel/schedules.py:2166
# 越靠前的 stage 预热越多；最后一个 stage 预热 0
num_warmup_microbatches = total_stages - current_stage - 1
num_warmup_microbatches = min(num_warmup_microbatches, num_microbatches)
num_microbatches_remaining = num_microbatches - num_warmup_microbatches

稳态循环是 1F1B 的灵魂——一次融合通信「把前向激活发给下游、同时收回上游传来的反向梯度」，紧接着立刻做一个反向:

Megatron-LM · pipeline_parallel/schedules.py:2269（steady 阶段）
for i in range(num_microbatches_remaining):
    output_tensor, _ = forward_step(...)                                   # 做 1 个前向 F
    output_tensor_grad = p2p_communicator.send_forward_recv_backward(...)  # 发激活 + 同时收回梯度
    input_tensor  = input_tensors.pop(0)                                   # 取出最早那个 micro-batch
    output_tensor = output_tensors.pop(0)
    input_tensor_grad = backward_func(input_tensor, output_tensor, output_tensor_grad, config)  # 做 1 个反向 B
    input_tensor = p2p_communicator.send_backward_recv_forward(...)        # 发梯度 + 同时收下一个激活

06再进一步:交错 1F1B 与 Zero-Bubble

1F1B 把显存解决了，但那段固定气泡 $(P-1)/(M+P-1)$ 还在。还能更狠:

  • 交错 1F1B（Interleaved，虚拟 stage）:不再让一张卡管「连续的 8 层」，而是管「不连续的几小块」（比如卡 0 管层 1–4 和层 17–20）。每张卡有 num_model_chunks 个虚拟 stage，流水线被切得更细，气泡随之 ∝ 1/chunks 缩小。代价是 p2p 通信次数翻倍。
  • Zero-Bubble:把「反向」拆成两件事——算激活的梯度（B，下游要用，必须及时）和算权重的梯度（W，自己更新用，可以晚点）。把不急的 W 塞进气泡里，几乎把气泡填到 0。

Megatron 用一个分发函数按 PP 度数和是否有虚拟 stage 自动选调度:

Megatron-LM · pipeline_parallel/schedules.py:142（get_forward_backward_func）
if pp_size > 1:
    if vp_size is not None:
        forward_backward_func = forward_backward_pipelining_with_interleaving       # 交错 1F1B
    else:
        forward_backward_func = forward_backward_pipelining_without_interleaving    # 普通 1F1B
else:
    forward_backward_func = forward_backward_no_pipelining

07stage 之间到底怎么传数据

PP 的通信不是 all-reduce 这种集合通信，而是点对点 send/recv(就像两个工位之间用传送带递东西)。Megatron 的 P2PCommunicator 封装了四个方向，外加两个稳态用的「边发边收」融合算子:

Megatron-LM · pipeline_parallel/p2p_communication.py:424 / :486
def recv_forward(...):   # 从上一 stage 收激活；如果自己是首 stage，没人传 → None
    if is_first_stage: input_tensor = None
    else: input_tensor, _, _ = self._communicate(recv_prev=True, recv_next=False, ...)

def send_forward(...):   # 把激活发给下一 stage；如果自己是尾 stage，不用发
    if not is_last_stage: self._communicate(tensor_send_next=output_tensor, ...)
# 另有 recv_backward / send_backward（传梯度），底层都走 torch.distributed 的 P2POp + batch_isend_irecv

08torchtitan:直接复用 PyTorch 的 pipelining

torchtitan 不自己写这套复杂调度，而是直接用 PyTorch 内置的 torch.distributed.pipelining。你给个 schedule 名字（GPipe / 1F1B / Interleaved1F1B / ZBVZeroBubble），它解析成对应的类:

torchtitan · distributed/pipeline_parallel.py:253（_build_pipeline_schedule）
schedule_class = get_schedule_class(parallelism.pipeline_parallel_schedule)   # 名字 → 调度类
n_microbatches = batch_size // microbatch_size
if issubclass(schedule_class, PipelineScheduleMulti):   # 多 stage（交错/zero-bubble）
    schedule = schedule_class(stages, n_microbatches=n_microbatches, loss_fn=loss_fn, ...)
else:                                                    # 单 stage（GPipe / 1F1B）
    schedule = schedule_class(stages[0], n_microbatches=n_microbatches, loss_fn=loss_fn, ...)

切模型时它真的把不属于本 stage 的子模块删掉/置 None，得到一个只含本 stage 几层的「瘦身版」模型:

torchtitan · distributed/pipeline_parallel.py:444（_split_module）
model = copy.deepcopy(whole_model)
for module_name, module_value in model.named_children():
    if isinstance(module_value, (nn.ModuleDict, nn.ModuleList, ...)):
        new_layers = ModuleList([layer for i, layer in enumerate(module_value) if i in indices_to_keep])
        setattr(model, module_name, new_layers)        # 只留本 stage 的层
    elif module_name not in modules_to_keep:
        setattr(model, module_name, None)              # 其它整模块置 None

训练循环里，只有首 stage 喂输入、尾 stage 收 loss，中间全由 schedule 的 step() 自动驱动:

torchtitan · trainer.py:661
if self.pp_has_first_stage:
    self.pp_schedule.step(inputs, target=targets, losses=losses, ...)   # 首 stage 喂输入
else:
    self.pp_schedule.step(target=targets, losses=losses, ...)
if self.pp_has_last_stage:                              # 只有尾 stage 才有 loss
    loss = torch.sum(torch.stack(losses)).to(self.device)

09对比与启用

Megatron-LM 手写调度

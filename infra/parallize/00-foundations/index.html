Module 0 · 基础铺垫

显存花在哪、卡之间怎么传数据

在学任何一种并行之前，先建立两块地基：一次训练 step 的显存账本，以及卡间通信的集合通信原语。后面 DP / FSDP / TP / PP / CP / EP 全是在这两件事上做文章。

🧮 16 字节/参数 🔁 all-reduce = reduce-scatter + all-gather 🔌 NVLink vs InfiniBand 📊 MFU / bubble / overlap

01为什么需要并行：两堵墙

单卡训练大模型会撞上两堵墙：

  • 显存墙：模型参数、梯度、优化器状态、激活值加起来远超单卡显存（H100 也只有 80GB）。一个 7B 模型光优化器状态就要 ~84GB——单卡装不下。
  • 算力墙：就算装得下，单卡算几万亿 token 要几十年。必须用成百上千张卡同时算。

并行的本质就是：把「模型状态」和「计算」拆开，摊到多张卡上。拆什么、怎么拆、拆完怎么把结果对齐——就是整本手册的内容。要理解拆什么，先看清显存到底花在哪。

02显存账本：一次训练 step 的显存去向

训练时显存由四部分构成，记参数量为 $P$：

组成是什么混合精度 + Adam 下的大小
参数 weights前向用的模型权重（bf16/fp16）$2P$ 字节
梯度 gradients反向算出的梯度（bf16/fp16）$2P$ 字节
优化器状态 optimizer statesfp32 master 权重 + Adam 一阶 $m$ + 二阶 $v$$4P + 4P + 4P = 12P$ 字节
激活值 activations前向中间结果，反向要用单独计算，∝ batch × seq × 层数

把前三项（模型状态）加起来，每个参数要 16 字节：

7B 模型：模型状态 = 7e9 × 16B ≈ 112 GB，单卡放不下。注意「优化器状态」占了 12/16 = 75%——这正是 ZeRO/FSDP 第一刀切它的原因。

🔑 记住这张账本

后面每种并行都对应「切账本里的某一项」：FSDP/ZeRO 切模型状态（参数+梯度+优化器状态）；TP/PP 切参数与对应计算；SP/CP 切激活值（按序列维）。看不懂某个并行省了什么时，回来对照这张表。

03通信原语：并行的「语言」

多卡协作靠集合通信（collective communication）。理解下面 6 个原语，后面所有并行的通信都能看懂。设有 4 张卡，每卡持有一份数据：

原语做什么典型用途
all-reduce各卡数据按元素求和（或取 max），结果每卡都得到完整一份DP 梯度同步、TP 行切层输出合并
reduce-scatter求和后每卡只保留 1/N 分片FSDP 梯度归约、SP 输出
all-gather把各卡的分片拼成完整一份，每卡都有FSDP 还原参数、SP 还原激活
broadcast一张卡的数据复制到所有卡初始化、PP 传 token 数
all-to-all每卡把自己的数据按目标切块，互相交换（转置）MoE 专家并行 dispatch/combine、Ulysses CP
send / recv点对点，一张卡发给另一张卡PP 跨 stage 传激活/梯度

🔑 最重要的恒等式

all-reduce = reduce-scatter + all-gather

这就是为什么 FSDP（reduce-scatter 梯度 + all-gather 参数）的通信量大约是 DDP（一次 all-reduce）的 1.5×——它把 all-reduce 拆成两半，中间插入了「只持有分片」的省显存窗口。后面会反复用到这个关系。

三个最常用的归约/聚合原语，以及把它们联系起来的恒等式。

这些原语在两个框架里都有最薄的封装。比如 Megatron 的 TP all-reduce 就是一行：

Megatron-LM · megatron/core/tensor_parallel/mappings.py:22
def _reduce(input_, group):
    """All-reduce the input tensor across model parallel group."""
    if group.size() == 1:
        return input_
    torch.distributed.all_reduce(input_.contiguous(), group=group)
    return input_

torchtitan 则把 loss 的归约包成 DTensor 友好的工具（注意它用的就是 functional collective 的 all_reduce）：

torchtitan · torchtitan/distributed/utils.py:30
def _dist_reduce(x, reduceOp, mesh, extra_pg):
    if isinstance(x, DTensor):
        # functional collectives 不支持 DTensor 输入，先取本地张量
        x = x.to_local()
    return float(funcol.all_reduce(x, reduceOp=reduceOp, group=mesh).item())

04硬件拓扑：带宽决定「谁放哪」

卡间带宽分层级，差距极大，这直接决定每种并行该放在哪一层：

层级互连量级带宽适合放
单机内（intra-node）NVLink / NVSwitch数百 GB/s ~ TB/sTP / EP（通信最密，每层都通信）
跨机（inter-node）InfiniBand / RoCE数十 ~ 数百 GB/s（远低于 NVLink）PP / DP / FSDP（通信稀疏或可与计算重叠）

💡 核心放置直觉 通信越频繁、越在关键路径上的并行（TP），越要放进高带宽的 NVLink 域（典型 TP≤8，正好一台机器）；通信稀疏、能和计算重叠的（PP / DP），可以容忍跨机的低带宽。这条原则会在 M7 nD 组合 里反复出现。

05衡量指标：怎么判断快不快

  • 吞吐 throughput：每秒处理多少 token（tokens/s）。最终目标。
  • MFU（Model FLOPs Utilization）：实际算力利用率 = 实测 FLOPs / 硬件峰值 FLOPs。40~50% 算不错，越高越好。
  • 气泡 bubble：流水线并行里 GPU 干等的空闲时间占比（M4 详谈）。
  • 通信-计算重叠 overlap：让集合通信和计算同时进行，把通信时间「藏」在计算后面。这是所有高效并行实现的命脉——后面会看到 FSDP 预取参数、DP 反向边算边 all-reduce、PP 用异步 p2p，全是为了 overlap。

06本手册地图：每种并行切账本里的什么

有了显存账本和通信原语，整本手册可以一句话概括成「切账本里的不同项，用不同原语对齐结果」：

模块切账本里的主要通信原语
M1 DP切数据（模型状态不切，冗余）all-reduce（梯度）
M2 FSDP/ZeRO切模型状态（参数+梯度+优化器状态）all-gather + reduce-scatter
M3 TP+SP切层内参数与计算；SP 切激活all-reduce / all-gather+reduce-scatter
M4 PP切层（不同层放不同卡）send / recv
M5 CP切序列维的激活ring p2p / all-to-all
M6 EP切专家（MoE）all-to-all

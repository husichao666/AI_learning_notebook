Module 6 · 专家并行

专家并行 EP：把 MoE 的专家放到不同卡

MoE 用稀疏激活把参数量和计算量解耦——专家很多但每个 token 只激活几个。EP 把不同专家放到不同卡，用 all-to-all 把 token 送到它该去的专家那儿，算完再送回来。

🧠 top-k 路由 + 稀疏激活 🔀 all-to-all dispatch / combine ⚖️ capacity + aux loss 🧩 与 TP/DP 组合

01先理解 MoE：稀疏激活

MoE（Mixture of Experts）把一个大 FFN 换成 $E$ 个并列的小 FFN（专家），外加一个路由器（router）。每个 token 只被路由到 top-k 个专家（通常 k=1 或 2），只有这些专家参与计算：

Megatron-LM · transformer/moe/moe_utils.py:780 / torchtitan · models/common/moe.py:225
# router 给每个 token 对 E 个专家打分，选 top-k
return torch.topk(scores, k=topk, dim=1, sorted=...)   # Megatron
_, selected_experts_indices = torch.topk(scores_for_choice, k=self.top_k, dim=-1)  # torchtitan

🔑 参数与计算解耦 256 个专家、top-2，意味着参数量是稠密模型的百倍，但每个 token 的 FLOPs 只增加 2 倍。这就是 MoE 能把模型做到万亿参数却还训得动的原因。代价：专家太多，单卡放不下全部专家——于是需要 EP。

02EP：专家切到各卡

专家并行（Expert Parallel, EP）把 $E$ 个专家平均分到 ep_size 张卡，每卡持 $E/\text{ep}$ 个：

Megatron-LM · transformer/moe/moe_layer.py:182
assert self.config.num_moe_experts % ep_size == 0
self.num_local_experts = self.config.num_moe_experts // ep_size      # 每卡持有的专家数
local_expert_indices_offset = ep_rank * self.num_local_experts
self.local_expert_indices = [local_expert_indices_offset + i for i in range(self.num_local_experts)]

torchtitan 用 DTensor 的 Shard(0)（专家维）表达同一件事：

torchtitan · distributed/expert_parallel.py:53
class ExpertParallel(ParallelStyle):
    def _partition_fn(self, name, mod, device_mesh):
        for param_name, param in mod.named_parameters(recurse=False):
            # 专家维 Shard(0)：每卡只存 E/ep 个专家的权重
            dist_param = nn.Parameter(distribute_tensor(param, device_mesh, [Shard(0)]))
            mod.register_parameter(param_name, dist_param)

03核心通信：all-to-all dispatch / combine

问题来了：token 在它原本所在的卡上，但它被路由到的专家在另一张卡上。怎么把 token 送过去？答案是 all-to-all——每张卡把自己的 token 按「该去哪张卡」分组，互相交换：

EP 的一来一回：dispatch（all-to-all 把 token 送到专家）→ 本地专家计算 → combine（all-to-all 送回）。

Megatron 的 all-to-all dispatcher：

Megatron-LM · transformer/moe/token_dispatcher.py:677（dispatch）/ :836（combine）
# dispatch：按 EP 维把 token 发到持有目标专家的卡
global_input_tokens = all_to_all(self.ep_group, permutated_local_input_tokens,
                                 self.output_splits, self.input_splits, ...)
# combine（反向）：把专家输出送回原 token 所在的卡（splits 互换）
permutated_local_input_tokens = all_to_all(self.ep_group, hidden_states,
                                 self.input_splits, self.output_splits, ...)

torchtitan 用 functional collective 的 all_to_all_single_autograd，结构完全一致：

torchtitan · models/common/token_dispatcher.py:312（dispatch）/ :434（combine）
routed_input = all_to_all_single_autograd(routed_input, output_splits_list, input_splits_list, self.ep_mesh)
# ... 本地专家计算 ...
routed_output = all_to_all_single_autograd(routed_output, metadata.input_splits, metadata.output_splits, self.ep_mesh)

💡 重叠技巧 两个框架都在 all-to-all 进行时，同时计算 shared expert（共享专家），把通信延迟藏在计算后面。

04负载均衡：capacity 与 aux loss

路由是模型自己学的，可能让某些「热门」专家挤满、冷门专家闲置——既浪费算力又撑爆显存。两道防线：

① capacity factor：硬上限 + 丢弃

给每个专家设容量上限，超出的 token 直接丢弃（或填充到等长以利分组 GEMM）：

Megatron-LM · transformer/moe/moe_utils.py:221
capacity = math.ceil((num_tokens / num_experts) * capacity_factor)   # 每专家容量
# 超过 capacity 的 token 按概率丢弃（capacity_factor=None 表示不丢）

② aux loss：软约束，把负载推向均匀

训练时加一个辅助损失 $\mathcal{L}_{aux} = E\sum_i f_i \cdot P_i$（$f_i$=分给专家 i 的 token 比例，$P_i$=路由器给它的平均概率），两者越均衡 loss 越小：

Megatron-LM · transformer/moe/moe_utils.py:142（switch_load_balancing_loss_func，源自 Switch Transformer）
aggregated_probs_per_expert = probs.sum(dim=0)
aux_loss = torch.sum(aggregated_probs_per_expert * tokens_per_expert) * (
    num_experts * moe_aux_loss_coeff / (topk * total_num_tokens * total_num_tokens))

05EP 与其它并行组合

EP 很少单独用，通常和 TP/DP 一起。专家内部还能再 TP（SequentialMLP 用专家 TP 组），dispatch 后接 TP all-gather、combine 前接 TP reduce-scatter。torchtitan 在 device mesh 上用独立的稀疏区 mesh 管理 EP，从 FSDP+TP「借」rank：$\text{efsdp} = \text{dp\_shard}\cdot\text{cp}\cdot\text{tp}/\text{ep}$（细节见 M7）。

torchtitan · models/llama4/parallelize.py:367（apply_moe_ep_tp，DeepSeek-V3 复用）
if ep_mesh is None:                    # EP 关：专家走 TP
    experts_mesh, experts_plan = tp_mesh, TensorParallel()
else:                                  # EP 开：专家走 EP
    experts_mesh, experts_plan = ep_mesh, ExpertParallel()
parallelize_module(module=transformer_block.moe.experts, device_mesh=experts_mesh, parallelize_plan=experts_plan)

06对比与启用

Megatron-LM

  • --expert-model-parallel-size N + --num-experts E
  • dispatcher：allgather(默认) / alltoall / flex
  • --moe-router-topk / --moe-aux-loss-coeff
  • --moe-expert-capacity-factor

torchtitan

  • expert_parallel_degree（config 层）
  • num_experts（模型 Config 字段）
  • ExpertParallel = Shard(0) 专家维
  • AllToAllTokenDispatcher（EP=1 退回本地）

📝 易错点（来自源码核对） ① torchtitan 的 ExpertParallel 本身不含 all-to-all，通信都在 token_dispatcher.py；② EP 用 Shard(0)（专家维），专家-TP 才是 Shard(1)/Shard(2)；③ Megatron 默认 dispatcher 是 allgather，大规模 EP 才切 alltoall。
